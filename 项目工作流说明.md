# 漫画项目工作流说明

## 目的

这份文档是一份通用漫画生产手册，用于指导任何维护者或 agent 在当前项目结构中完成一部漫画从构思到出图的全过程。

本文档不绑定任何具体作品设定，不记录具体人物、剧情或世界观，只记录方法、顺序、格式、技术约束和验收标准。

## 总流程

本项目的标准生产路径固定为：

1. 故事
2. 人物设定
3. 分镜
4. 详细分镜描述
5. 提示词
6. 生成图片

这六步必须按顺序推进，不要跳步。

原因很简单：

- 没有故事，分镜就没有因果和节奏。
- 没有人物设定，提示词就无法稳定生成同一角色。
- 没有明确分镜，图片只会是零散插图，不是漫画。
- 没有详细分镜描述，提示词会失去镜头信息。
- 没有提示词规范，批量生成会失控。
- 没有正确导入规则，生成的图片会在排版中变形或失真。

## 1. 故事

故事阶段只解决结构问题，不解决画面问题。

在这个阶段，必须先明确以下内容：

- 主题是什么。
- 主角要达成什么。
- 核心冲突来自哪里。
- 故事的转折点在哪里。
- 情绪最高点在哪里。
- 这一话或这一卷的收束点在哪里。

建议先写一份短纲，再扩成页级结构。

短纲至少回答：

- 开场要做什么。
- 中段推进什么。
- 最终变化是什么。

如果故事阶段没有先完成，不要提前写提示词，也不要提前出图。

## 2. 人物设定

人物设定的目标不是“写得好看”，而是“让角色在几十页图像里保持一致”。

每个主要角色至少需要三层设定。

### 2.1 角色基准

用于固定人物识别信息。

必须明确：

- 年龄感
- 性别感
- 身高和体型
- 脸型
- 五官特征
- 发型和发色
- 肤色
- 气质关键词
- 默认姿态和表情倾向

### 2.2 外观变体

用于区分不同场景、不同时间、不同状态下的服装与外观。

每套外观都应明确到可直接用于文生图：

- 服装类别
- 版型
- 面料
- 主色和辅助色
- 长度
- 配饰
- 鞋子
- 必须排除的元素

不要写“普通私服”“常规校服”这种模糊词，必须写成可执行描述。

### 2.3 能力或道具规则

如果角色有特殊能力、武器、职业制服或固定道具，要单独写规则。

至少说明：

- 是否会改变服装
- 是否会出现固定光效或特效
- 哪些视觉元素是稳定出现的
- 哪些视觉元素不允许出现

这一层的目标是避免后期提示词在不同页之间自相矛盾。

## 3. 分镜

分镜阶段解决的是页面节奏，不是句子润色。

在这个阶段，先把故事拆成页。

每一页至少要有：

- 页码
- 页面功能
- 情绪作用
- 翻页点或落点
- 这一页采用什么分镜拓扑

分镜设计时应遵守基本创作规则：

- 重要信息要有足够大的画面承载。
- 强情绪页可以使用单页大格。
- 普通推进页不要切得过碎。
- 不要整本都重复同一种上下二分或左右二分。
- 版式变化要服务叙事，不要只为了炫技。

在本项目里，分镜最终需要写回 MangaMaker，因此每页的 panel 数量、形状和拓扑必须可落地。

## 4. 详细分镜描述

这一阶段是连接“分镜”与“提示词”的桥梁。

每一个分镜都必须写成明确的、可视化的描述，而不是概括性的摘要。

每格至少要说明：

- 在页面中的位置
- 占据多少面积
- 镜头远近
- 视角
- 画面主体是谁
- 主体在做什么
- 表情和身体状态
- 环境中最重要的可见信息
- 与前后格的叙事关系

推荐每格使用统一结构：

```text
位置 + 面积 + 镜头类型 + 角色/主体 + 动作 + 表情/情绪 + 环境要点 + 叙事作用
```

这个阶段允许写对白、动作和情绪，但必须保证：

- 即使删掉对白，这一格也仍然能被画出来。
- 不能把不可视的解释性信息当成画面内容。
- 不要只写“他很震惊”，而要写出可视的震惊表现。

## 5. 提示词

提示词阶段的任务不是“再讲一遍剧情”，而是把详细分镜描述转成适合图像模型的自然语言输入。

每个分镜的 `description` 都必须对应一条独立提示词。

### 5.1 提示词结构

建议按以下顺序组织：

```text
全局风格 + 场景环境 + 角色外观 + 镜头构图 + 当前动作 + 情绪细节 + 负面约束
```

### 5.2 提示词规则

必须遵守：

- 使用自然语言，不要写成项目内备注。
- 每一条提示词都必须是完整句式或完整自然语言片段，不能只是标签堆叠。
- 每一条提示词都必须是完全独立的，不能依赖上一格、下一格或页级上下文才能成立。
- 每一条提示词都必须在自身内部完整描述该图要出现的内容。
- 每一条提示词都必须对应一个静态 shot，一个冻结的单一画面，而不是连续动作过程说明。
- 保持同一角色名称稳定一致。
- 使用完整且可识别的人物命名。
- 服装、发型、体型、颜色等关键信息不能漂移。
- 不要把对白直接写进图片提示词。
- 不要要求模型生成对白框、字效或排版边框。
- 不要用“漫画格本身”当作生成对象。

### 5.3 语言要求

如果批量生成依赖统一模型，提示词语言必须统一。

当前项目约束是：

- `description` 使用英文。
- 不混入中文。
- 不混入源文档式注释。

### 5.4 提示词与分镜的关系

一个 panel 的提示词只能负责这一格的画面，不负责整页版式。

也就是说：

- 分镜负责 panel 怎么切。
- 提示词负责 panel 里画什么。

不要把这两个层级混在一起。

### 5.5 静态画面约束

提示词应描述“这一刻看见了什么”，而不是“接下来发生什么”或“动作如何连续推进”。

推荐写法：

- 角色当前姿态
- 当下可见的动作结果
- 当前表情
- 当前空间关系
- 当前环境状态

应尽量避免：

- begin to
- continue to
- about to
- then
- after that
- sequence of actions

如果一个分镜里有多个可见信息，应把它们组织成“同一张图里同时成立的视觉元素”，而不是写成时间顺序。

## 6. 生成图片

这一阶段的任务是把提示词变成图像，并正确导入每个 panel。

必须分成两个部分理解：

1. 图像生成
2. 图像导入

### 6.1 图像生成规则

生成的是“正常的单张彩图”，不是整页漫画版。

因此提示词里不应该要求：

- panel border
- comic frame
- speech bubble
- sound effect text
- page layout

模型只负责生成单格内容。

### 6.2 工作流参数来源

如果使用 Forge 进行批量生成，采样器、调度器、CFG、步数、分辨率等工作流参数必须先读取前端当前配置，再应用模型专属覆盖规则，不能直接跳过前端状态盲目生成。

推荐读取顺序：

1. 前端 `ui-config.json`
2. 前端 `config.json`
3. 运行中接口 `/sdapi/v1/options`

至少要读取：

- sampler
- scheduler
- steps
- CFG
- distilled CFG
- seed
- batch size
- batch count
- width
- height

如果前端启用了高分辨率修复，也要同步读取：

- hires steps
- denoising strength
- upscale by
- resize width / height
- hires sampler
- hires scheduler
- hires CFG
- hires distilled CFG

规则是：

- 前端先提供当前工作流参数。
- 模型专属兼容规则在读取后覆盖特定参数。
- 分镜提示词决定画面内容。

不要把这两层职责混淆。

### 6.3 anima 模型固定规则

如果当前生成使用的是 anima 工作流，则以下规则优先级高于前端当前 sampler、scheduler 和分辨率控件值：

- sampler 必须使用 `ER-SDE`
- scheduler 必须使用 `Simple`
- 分辨率只能使用以下三种之一：
  - `768 x 1024`
  - `1024 x 1024`
  - `1024 x 768`

推荐按 panel 纵横比选择最接近的一个固定分辨率：

- 竖构图使用 `768 x 1024`
- 接近正方形的构图使用 `1024 x 1024`
- 横构图使用 `1024 x 768`

如果存在会让输出分辨率跳出这个固定集合的附加流程，例如某些高分辨率修复设置，应在 anima 工作流下禁用。

### 6.4 图像导入规则

图片导入到 panel 时，只允许：

- 等比缩放
- 裁切
- 平移 viewBox

不允许：

- 单独拉伸宽度
- 单独压缩高度
- 为了填满 panel 而做一维变形

正确做法是：

- 根据 panel 的纵横比计算一个匹配比例的 `viewBox`
- 如果源图更宽，则左右裁切
- 如果源图更高，则上下裁切
- 默认使用居中裁切

这一规则非常重要，因为 MangaMaker 的 panel 渲染本身会按宽高分别映射。如果导入时直接把整张源图完整塞进不同比例的 panel，就会产生拉伸。

### 6.5 对白与文字层

推荐的顺序是：

1. 先完成纯图像生成和导入
2. 再单独生成或回填对白气泡
3. 最后再处理字体、字号、气泡位置和尾巴指向

不要把对白本身交给文生图模型。

## 当前仓库中的落地文件

这份手册是通用方法，但当前仓库已经有一套具体落地结构。

常用入口如下：

| 文件 | 作用 |
| --- | --- |
| `漫画大纲.source.md` | 源内容文档，应优先从这里维护故事、人物和分镜。 |
| `漫画大纲.md` | 与当前项目同步的展示版文档。 |
| `MangaMaker/scripts/build_yukino_project_template.mjs` | 把文档转换成 MangaMaker 项目模板。 |
| `MangaMaker/scripts/generate_yukino_panel_assets.mjs` | 读取前端配置、生成图片、写回 panel，并回填气泡。 |
| `MangaMaker/src/generated/yukinoProject.json` | 当前生成出的项目模板。 |
| `MangaMaker/src/bootstrap/syncYukinoProject.ts` | 启动时把模板写入浏览器项目库。 |
| `MangaMaker/src/main.tsx` | 应用入口。 |

## 推荐执行顺序

### 只改故事、人物或分镜

1. 修改 `漫画大纲.source.md`
2. 运行模板生成脚本
3. 检查 `漫画大纲.md`
4. 检查 `yukinoProject.json`
5. 构建并刷新 MangaMaker

### 重新生成分镜图片

1. 确认前端 Forge 当前配置正确
2. 运行图片生成脚本
3. 将图片写回 panel
4. 验证图像没有被拉伸
5. 构建并刷新 MangaMaker

### 仅修复导入参数，不重跑图片

如果已有图片是可用的，只需要重算 panel 的导入裁切参数，可以复用已有图片并只重写项目数据。

## 验收清单

每次交付前至少检查以下内容：

- 故事结构已经先于分镜确定。
- 人物设定足够支持稳定出图。
- 页级分镜能清楚表达节奏。
- 每个分镜都有详细描述。
- 每个 panel 的 `description` 都是自然语言提示词。
- 提示词不混入对白文本。
- 提示词语言统一。
- 提示词是自然语言、独立、完整、静态的单图描述。
- 生成参数来自前端工作流，而不是脚本硬编码。
- anima 工作流使用 `ER-SDE`、`Simple` 和固定三种分辨率之一。
- 导入后的图像没有单轴拉伸。
- 对白框在图像生成后再独立处理。
- MangaMaker 构建通过。
- 刷新后可以在当前浏览器 profile 中看到更新结果。

## 维护建议

- 不要直接长期手改生成产物。
- 不要跳过“详细分镜描述”这一步。
- 不要把故事梗概直接当成提示词。
- 不要让模型同时承担“画图”和“排版整页”两个任务。
- 不要在不知道前端当前配置的情况下直接批量跑图。
- 如果某一步结果不稳定，应回到上一层修正，而不是只在最后一步硬补。
