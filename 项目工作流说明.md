# 项目工作流说明

## 目的
本文档定义当前仓库中“从故事到成品项目”的标准漫画生产流程。
它服务于维护者、脚本作者和 agent，要求在当前仓库结构里，把一部作品从构思、分镜、图像生成、导入排版，一直落到 `projects/` 持久化目录中。

本文档不绑定特定 IP，不规定必须创作哪一部作品；它规定的是流程、顺序、格式、落盘方式和验收标准。

## 总流程
当前项目的标准生产路径固定为六步：

1. 故事
2. 人物设定
3. 分镜
4. 详细分镜描述
5. 提示词
6. 生成图片并导入项目

这六步必须按顺序推进，不能跳步。

原因：

- 没有故事，分镜就没有节奏与因果。
- 没有人物设定，跨页生成时角色会漂移。
- 没有分镜，输出只会是散图，不是漫画页面。
- 没有详细分镜描述，提示词会丢镜头信息。
- 没有提示词规范，批量生成会失控。
- 没有正确导入与持久化规则，图片会在排版中变形，或者工作成果不会即时落盘。

## 项目类型约束
当前仓库支持两种项目类型：

- `manga`
- `cg`

约束如下：

- `manga` 默认页面尺寸：`1200 x 1700`
- `cg` 默认页面尺寸：`1200 x 1600`
- 创建项目时必须明确指定项目类型
- 项目类型可以在项目创建后修改，但修改后只影响后续默认行为，不自动重写旧页面
- `cg` 项目中，粘贴图片时默认创建新的 `1200 x 1600` 全 stage 分镜

## 持久化约束
所有重要改动都必须即时落盘，不能只停留在内存或浏览器缓存中。

网页运行时：

- 项目写入仓库根目录 `projects/`
- 每个项目使用单独文件夹：`projects/<sanitized-project-title>/`
- 项目数据写入：`projects/<sanitized-project-title>/project.json`
- 导入或粘贴的图片写入：`projects/<sanitized-project-title>/assets/`
- 重要操作后必须立即落盘，包括：项目创建、页面创建、分镜创建、图片导入、图片粘贴、文字修改、气泡修改、页面删除等

桌面运行时：

- 项目写入本地未跟踪 `projects/`
- 当前桌面运行时仍使用项目 id 作为文件夹名：`projects/<project-id>/`
- 项目数据写入：`projects/<project-id>/project.json`
- 导入或粘贴的图片写入：`projects/<project-id>/assets/`

## 六个阶段的具体要求
### 1. 故事
故事阶段先解决结构问题，不解决提示词问题。

至少要明确：

- 主题是什么
- 主角想达成什么
- 核心冲突来自哪里
- 转折点在哪里
- 情绪最高点在哪里
- 本话或本卷收束在哪里

推荐先写一份短纲，再扩成页级结构。

### 2. 人物设定
人物设定的目标不是“写得漂亮”，而是“保证角色跨页稳定”。

每个主要角色至少要写清：

- 年龄感
- 体型与身高
- 脸型与五官特征
- 发型与发色
- 服装版本
- 材质、主色与辅助色
- 常见姿态与表情倾向
- 必须避免出现的漂移项

如果角色有武器、特殊能力、职业制服或固定道具，必须单独写规则。

### 3. 分镜
这一阶段要把故事拆成页，并确保能真实落地到 MangaMaker。

每一页至少要有：

- 页码
- 页面功能
- 情绪作用
- 翻页落点
- 分镜数量与版式思路

要求：

- 重要信息必须获得足够大的画面承载
- 版式变化必须服务叙事，不为炫技而变
- 每页最终都要能被转换成实际 `panel` 数据

### 4. 详细分镜描述
详细分镜描述是“页级分镜”和“单格提示词”之间的桥。

每一格至少写清：

- 在页面中的位置
- 占比或面积
- 镜头远近
- 视角
- 主要主体是谁
- 主体正在做什么
- 表情与身体状态
- 环境中的关键信息
- 它在前后叙事中的作用

推荐统一结构：

```text
位置 + 面积 + 镜头类型 + 主体 + 动作 + 情绪/表情 + 环境要点 + 叙事作用
```

### 5. 提示词
提示词的目标不是“再讲一遍剧情”，而是把每个 panel 的详细描述转换成适合图像模型的单张画面输入。

要求：

- 每个 panel 的 `description` 对应一条独立提示词
- 使用自然语言，而不是内部备注
- 每条提示词必须自包含，不能依赖上一格或下一格
- 每条提示词必须描述一个静止的单张画面，而不是动作序列
- 不要把对白直接塞进图像提示词
- 不要要求模型直接生成对话框、页面边框或排版元素

当前仓库约束：

- `description` 使用英文
- 提示词语言保持统一
- 角色命名、服装、体型和关键视觉信息必须稳定

### 6. 生成图片并导入项目
这一步分成两部分：

1. 图像生成
2. 图像导入与项目写入

#### 6.1 图像生成规则
生成的是“单格内容图”，不是整页漫画版式。

因此提示词里不应要求：

- panel border
- comic frame
- speech bubble
- sound effect text
- page layout

#### 6.2 Forge 工作流参数来源
如果通过 Forge 批量生成，参数必须先读取当前前端状态，再应用模型专属覆盖规则。

推荐读取顺序：

1. 前端 `ui-config.json`
2. 前端 `config.json`
3. 运行中接口 `/sdapi/v1/options`

至少要读取：

- sampler
- scheduler
- steps
- CFG
- distilled CFG
- seed
- batch size
- batch count
- width
- height

#### 6.3 anima 工作流固定规则
如果当前使用的是 anima 工作流，优先级高于前端当前 sampler、scheduler 和分辨率控件值的规则如下：

- sampler 必须使用 `ER-SDE`
- scheduler 必须使用 `Simple`
- 分辨率只能使用下列三种之一：
  - `768 x 1024`
  - `1024 x 1024`
  - `1024 x 768`

推荐按 panel 的纵横比选择最接近的一档。

#### 6.4 导入规则
把图片放进 panel 时，只允许：

- 等比缩放
- 裁切
- 平移 viewBox

不允许：

- 单独拉伸宽度
- 单独压缩高度
- 为了填满 panel 而强行改变图片比例

正确做法：

- 根据 panel 的纵横比计算匹配的裁切 viewBox
- 源图更宽时左右裁切
- 源图更高时上下裁切
- 默认使用居中裁切

#### 6.5 对白与文字层
推荐顺序：

1. 先完成纯图片生成和导入
2. 再回填对白与气泡
3. 最后处理字体、字号、方向、气泡位置和尾巴指向

## 当前仓库中的落地文件
当前仓库里，已维护的示例流程以 `PsychoPass-GreyNoise` 为主。

核心文件：

- `docs/projects/psychopass-grey-noise.source.md`：故事、角色与分镜源文档
- `docs/projects/psychopass-grey-noise.md`：对外展示版文档
- `scripts/psychopassGreyNoiseData.mjs`：项目数据与路径约定
- `scripts/create_psychopass_grey_noise_project.mjs`：生成项目、调用 Forge、写入资源与项目 JSON
- `src/generated/psychopassGreyNoiseProject.json`：生成出的模板项目 JSON
- `projects/PsychoPass-GreyNoise/project.json`：最终落盘项目数据
- `projects/PsychoPass-GreyNoise/assets/`：最终落盘图片资源

执行入口通常是：

```bash
node scripts/create_psychopass_grey_noise_project.mjs
```

## 推荐执行顺序
### 只改故事、角色或分镜
1. 修改 `docs/projects/*.source.md`
2. 同步更新展示文档 `docs/projects/*.md`
3. 修改数据脚本（例如 `scripts/psychopassGreyNoiseData.mjs`）
4. 运行项目生成脚本
5. 检查 `src/generated/*.json`
6. 检查 `projects/<ProjectTitle>/project.json`
7. 检查 `projects/<ProjectTitle>/assets/`

### 重新生成图像
1. 确认 Forge 当前配置正确
2. 运行项目生成脚本
3. 检查输出资源是否已写入 `projects/<ProjectTitle>/assets/`
4. 检查 `project.json` 中的图片 `src` 是否指向 `/projects/<ProjectTitle>/assets/...`
5. 在 MangaMaker 中打开项目进行排版复核

### 仅修正导入参数，不重跑图像
1. 保留已生成图片
2. 只重写 panel 的导入裁切数据
3. 重写项目 JSON
4. 验证图像未发生变形

## 验收清单
每次交付前至少检查：

- 六个阶段按顺序完成，没有跳步
- 人物设定足够支撑跨页稳定出图
- 页级分镜能落成真实 panel 数据
- 每个 panel 都有清晰的详细描述
- 每个 panel 的 `description` 都是独立、完整、自然语言的英文提示词
- 提示词没有直接混入对白排版要求
- anima 工作流遵守 `ER-SDE`、`Simple` 和固定三档分辨率规则
- 导入后的图像没有单轴拉伸
- 对白与气泡在图像导入之后单独处理
- 项目已真实写入 `projects/` 文件夹
- 重要操作后的改动会立即落盘
- MangaMaker 构建与测试通过

## 维护说明
当前仓库仍保留了一些旧的 Yukino 脚本和生成物，例如：

- `scripts/build_yukino_project_template.mjs`
- `scripts/generate_yukino_panel_assets.mjs`
- `src/generated/yukinoProject.json`

这些文件现在属于历史样例或迁移遗留，不应再被当作“当前主流程”的唯一依据。
新的文档、脚本和项目流程应优先参考当前维护中的 `psychopass` 链路与 `projects/` 即时持久化规则。
