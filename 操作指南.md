# 操作指南 / User Guide

This file defines how a human must be able to use MangaMaker.

本文档定义人类用户必须能够如何使用 MangaMaker。

## 1. 启动 / Start
Web shared mode:

网页版分享模式：
```bash
pnpm install
pnpm dev
```

Expected web behavior:

网页版预期行为：

- The terminal prints a local URL.
- Share mode is enabled by default.
- A public share link is printed in the terminal when the configured share provider succeeds.

Optional ngrok shared mode:

```bash
pnpm dev -- --share-provider ngrok
```

- 终端会打印本地访问地址。
- 默认启用分享模式。
- 当隧道建立成功后，终端会打印一个 72 小时有效的 Gradio 分享链接。

Web local-only mode:

网页版仅本地模式：
```bash
pnpm dev -- --no-share
```

Desktop:

桌面版：
```bash
pnpm tauri dev
```

## 2. 首次进入 / First Run
The startup welcome flow must let a new user:

首次进入时，新用户必须能完成以下流程：

1. See existing projects on the welcome screen with first-page thumbnails.
2. Create a project by entering a title and choosing `manga` or `cg`.
3. Open an existing project.
4. Delete a project from the welcome screen through the project card context menu.
5. Enter the editor and immediately see the page list, canvas area, right Inspector, and sticky top ribbon.
6. Understand the next step from the side panels rather than from a banner above the workspace.

1. 在欢迎页看到已有项目及其第一页缩略图。
2. 输入标题并选择 `manga` 或 `cg` 来创建项目。
3. 打开已有项目。
4. 通过项目卡片右键菜单删除项目。
5. 进入编辑器后立刻看到页面列表、画布区、右侧 Inspector 和会悬浮的顶部 ribbon。
6. 通过侧边区域理解下一步，而不是依赖 workspace 上方横幅。

Required behavior:

必需行为：

- Project type is part of project creation, not a hidden advanced option.
- The current project title and project type remain editable from the sidebar after project creation.
- Changing project type affects future defaults such as new-page size and image-paste behavior; it does not need to rewrite existing pages automatically.
- The welcome flow must not require reading code.

- 项目类型必须属于创建项目的基础流程，不能藏在高级选项里。
- 创建后，当前项目标题和项目类型仍可在侧边栏中修改。
- 修改项目类型会影响后续默认行为，例如新页面尺寸和图片粘贴行为；不要求自动重写已有页面。
- 欢迎流程不能要求用户阅读代码才能继续。

## 3. 编辑区结构 / Editor Regions
From outside to inside, the editor is organized as follows:

从外到内，编辑器区域分为：

1. **Editor shell / 编辑器外壳**: the full editing screen, including top ribbon, left sidebar, center canvas column, and right Inspector.
2. **Canvas container / 画布容器**: the scrollable center viewport that shows the current editable scene.
3. **Workspace / 工作区**: the larger editable plane inside the canvas container. Objects may extend outside the page while staying inside this workspace.
4. **Comic page / 漫画页面**: the exportable page inside the workspace. This is the page that PNG/PDF export uses.

1. **Editor shell / 编辑器外壳**：完整编辑屏幕，包含顶部 ribbon、左侧栏、中间画布列和右侧 Inspector。
2. **Canvas container / 画布容器**：中间可滚动的视口，用来显示当前可编辑场景。
3. **Workspace / 工作区**：画布容器里的更大编辑平面。对象可以越过页面，但必须留在这个工作区内。
4. **Comic page / 漫画页面**：工作区内部可导出的页面区域。PNG/PDF 导出以它为准。

Required behavior:

必需行为：

- The comic page fits inside the visible canvas frame at the default fit view.
- The workspace is visibly larger than the page, about 8x page area.
- The canvas container may be intentionally taller than the viewport and require vertical scrolling for precision placement.
- The top ribbon remains visible while scrolling.
- Crossing 100% zoom must remain continuous and must not suddenly resize the canvas container itself.

- 默认适配视图下，漫画页面必须完整落在可见画布框内。
- 工作区必须明显大于页面，面积约为页面面积的 8 倍。
- 画布容器可以故意比视口更高，并允许通过垂直滚动进行精细摆放。
- 顶部 ribbon 在滚动时必须保持可见。
- 缩放跨过 100% 时必须连续，不能让画布容器本身突然变大。

## 4. 页面操作 / Page Operations
The user must be able to:

用户必须能够：

1. Add a page.
2. Duplicate a page.
3. Delete a page.
4. Reorder pages.
5. Switch the current page from the page list.
6. Change page background color from the ribbon.
7. Use `Home` to return to the welcome screen.
8. Use `Save` to manually persist the current project immediately.

1. 新增页面。
2. 复制页面。
3. 删除页面。
4. 调整页面顺序。
5. 从页面列表切换当前页面。
6. 在 ribbon 中修改页面背景色。
7. 用 `Home` 返回欢迎页。
8. 用 `Save` 立即手动保存当前项目。

Required behavior:

必需行为：

- Page cards expose duplicate, move, and delete actions through the page context menu.
- Page deletion is immediate, shows no confirmation popup, and remains undoable with `Ctrl/Cmd+Z`.
- The page list always shows order-based numbering, not duplicate-style names.

- 页面卡片必须通过右键菜单提供复制、移动和删除操作。
- 删除页面必须立即执行，不弹确认框，但仍可通过 `Ctrl/Cmd+Z` 撤销。
- 页面列表显示的页面名称必须始终按顺序编号，而不是显示副本式名称。

## 5. 分镜与图片 / Panels and Images
The user must be able to:

用户必须能够：

1. Create, move, resize, and delete panels.
2. Convert a panel to a polygon and edit its vertices.
3. Import or replace an image for a panel.
4. Select a panel with an image and see the bound source image outside the mask.
5. Drag the visible image crop live and zoom it with the mouse wheel.
6. Move panel vertices or resize handles without dragging the bound image through stage space.
7. Edit a non-rendered panel description in the right Inspector.

1. 创建、移动、缩放和删除分镜。
2. 把分镜改成多边形并编辑顶点。
3. 为分镜导入或替换图片。
4. 选中带图分镜后，在遮罩外看到绑定源图。
5. 实时拖拽可见裁切区域，并通过滚轮缩放它。
6. 调整分镜顶点或缩放手柄时，不把绑定图片在 stage 中一起拖走。
7. 在右侧 Inspector 中编辑不参与渲染的分镜描述。

Required behavior:

必需行为：

- A panel is a crop window, not a whole-image scale container.
- Selecting a panel with an image reveals the bound source image and highlights the cut region.
- Left-drag on the semi-transparent image pans the crop.
- Left-drag on the panel border still moves the panel itself.
- Image crop updates must be visible live before mouse release.
- Dragging a panel does not auto-select it at drag end; a pure click selects it.
- Right-click inside the editing surface opens the custom context menu instead of the browser menu.
- Context menus for panel, text, and bubble support layer up/down actions.
- When selected content crosses the page edge, the page boundary must remain readable as a dashed overlay above it.

- 分镜本质上是裁切窗口，不是整图缩放容器。
- 选中带图分镜时，必须显示绑定源图并高亮当前裁切范围。
- 在半透明图片上左键拖拽时，必须平移裁切区域。
- 在分镜边框上左键拖拽时，仍然必须移动分镜本体。
- 图片裁切变化必须在松手前实时可见。
- 拖拽分镜结束后不能自动选中它；只有纯点击才会选中。
- 编辑区右键必须弹出自定义菜单，不能退回浏览器默认菜单。
- 分镜、文字和气泡的右键菜单都必须支持图层上移/下移。
- 当已选中的对象压到页面边界时，页面边界必须以虚线叠加在对象上方保持可读。

### 图片粘贴规则 / Image Paste Rules
For pasted clipboard images in `manga` projects:

`manga` 项目中的图片粘贴规则：

1. If a panel is selected, paste into that panel.
2. Otherwise, if the pointer is hovering a panel, paste into that panel.
3. Otherwise, create a new panel near the pointer and place the image there.

1. 如果当前有选中的分镜，优先粘贴进该分镜。
2. 否则，如果鼠标悬停在某个分镜上，粘贴进该分镜。
3. 否则，在指针附近新建一个分镜，并把图片放进去。

For pasted clipboard images in `cg` projects:

`cg` 项目中的图片粘贴规则：

- Every pasted image creates a new `1200 x 1600` full-stage panel by default.
- The panel is centered on the page as far as the current page size allows.
- This behavior takes priority over selected-panel and hovered-panel targeting.

- 每次粘贴图片都默认创建一个新的 `1200 x 1600` 全 stage 分镜。
- 该分镜会在当前页面尺寸允许的前提下尽量居中。
- 该行为优先级高于“粘贴进选中分镜”与“粘贴进悬停分镜”。

## 6. 文字与气泡 / Text and Bubbles
The user must be able to:

用户必须能够：

1. Add, edit, move, resize, and delete text boxes.
2. Switch text between horizontal and vertical layout.
3. Change font, size, direction, horizontal alignment, and vertical alignment from the right Inspector.
4. Add, edit, move, resize, and delete speech bubbles.
5. Switch among the supported bubble types.
6. Adjust bubble border width, fill color, border color, and type-specific parameters.
7. Use text boxes for dialogue lettering inside or near bubbles; adjust text direction and alignment from the right Inspector.
8. Drag a non-explosion bubble tail tip.
9. Drag the connection point between the tail and the bubble body.

1. 创建、编辑、移动、缩放和删除文本框。
2. 在横排和竖排之间切换文本布局。
3. 在右侧 Inspector 中修改字体、字号、方向、水平对齐和垂直对齐。
4. 创建、编辑、移动、缩放和删除气泡。
5. 在支持的气泡类型之间切换。
6. 调整气泡边框宽度、填充颜色、边框颜色和各类型专属参数。
7. 使用文字框承载气泡内或气泡附近的对白文字，并在右侧 Inspector 中调整文字方向和对齐。
8. 拖拽非爆炸气泡的尾巴尖端。
9. 拖拽尾巴与气泡本体的连接点。

Required behavior:

必需行为：

- New text boxes default to vertical text.
- Dialogue text is stored as text boxes rather than bubble fields, so it follows the normal new-text vertical default.
- Continuous numeric bubble controls use sliders in the Inspector.
- Non-explosion bubble tail editing must keep one continuous outer outline after moving the tail tip or tail base.
- The bubble interior must not reveal a separate tail border line.
- Explosion bubbles support per-spike editing instead of a separate tail.

- 新建文本框默认竖排。
- 对白文字保存为文字框而不是气泡字段，因此遵守普通新建文字默认竖排规则。
- Inspector 中连续可调的气泡数值必须使用滑杆。
- 非爆炸气泡在拖动尾尖或尾根连接点后，仍必须保持连续外轮廓。
- 气泡内部不能露出独立的尾巴边线。
- 爆炸气泡使用尖刺编辑，而不是独立尾巴。

## 7. 持久化 / Persistence
The user must be able to trust that important work is preserved without continuous disk writes.

用户必须能够相信：重要工作不会因为返回首页、切换项目或关闭页面而丢失，同时编辑过程中不进行高频落盘。

Required behavior:

必需行为：

- Mutating operations mark the current project as having unsaved changes.
- Manual `Save` persists the latest project immediately and remains available from the ribbon.
- Returning Home, creating/opening/restoring another project, and closing or hiding the page must save dirty work first.
- Imported or pasted images are materialized to the target project assets when the import/paste happens, so later project saves reference stable asset paths.

- 修改项目的操作必须把当前项目标记为存在未保存改动。
- ribbon 中仍然必须提供手动 `Save`，点击后立即保存最新项目。
- 返回 Home、创建/打开/恢复其他项目、关闭或隐藏页面时，必须先保存未保存改动。
- 导入或粘贴的图片必须在操作发生时写入目标项目资源目录，这样后续项目保存会引用稳定资源路径。

Web runtime:

网页版：

- Projects persist under repository-root `projects/`.
- Each project uses its own folder named from the sanitized project title.
- If a folder name conflicts, a unique suffix may be added.
- Project data persists to `projects/<sanitized-project-title>/project.json`.
- Imported or pasted images persist to `projects/<sanitized-project-title>/assets/`.

- 项目会持久化到仓库根目录的 `projects/`。
- 每个项目使用一个按项目标题清洗后的独立文件夹。
- 如果文件夹名冲突，可以自动附加唯一后缀。
- 项目数据写入 `projects/<sanitized-project-title>/project.json`。
- 导入或粘贴的图片写入 `projects/<sanitized-project-title>/assets/`。

Desktop runtime:

桌面版：

- Projects persist under a local untracked `projects/` folder.
- The current desktop runtime uses the project id as the folder name.
- Project data persists to `projects/<project-id>/project.json`.
- Imported or pasted images persist to `projects/<project-id>/assets/`.

- 项目会持久化到本地未跟踪的 `projects/` 文件夹。
- 当前桌面版运行时使用项目 id 作为文件夹名。
- 项目数据写入 `projects/<project-id>/project.json`。
- 导入或粘贴的图片写入 `projects/<project-id>/assets/`。

## 10. 人类验收 / Human Acceptance
Minimum human acceptance:

最低人类验收标准：

1. A new user can create a `manga` or `cg` project without outside help.
2. The user can understand that the center scrollable area is the canvas container, the larger editable plane is the workspace, and the page inside it is the exportable comic page.
3. The user can finish one basic page within minutes.
4. The user can discover project deletion, page deletion, project type change, and save without reading code.
5. The user can understand that a panel is a crop window rather than a whole-image scaler.
6. The user can see live crop updates while dragging a selected panel image.
7. The user can paste images into the selected or hovered panel in manga mode, and into a new full-stage panel in CG mode.
8. The user can adjust text typography and bubble shape/style from the right Inspector.
9. The user can drag both the bubble tail tip and the tail-base connection point without breaking the bubble outline.
10. The user can trust that closing the app after an important action will not lose work.

1. 新用户能在无外部帮助的情况下创建 `manga` 或 `cg` 项目。
2. 用户能理解：中间可滚动区域是画布容器，更大的编辑平面是工作区，内部可导出的页面是漫画页面。
3. 用户能在几分钟内完成一页基础页面。
4. 用户无需阅读代码，就能找到项目删除、页面删除、项目类型切换和保存。
5. 用户能理解分镜是裁切窗口，而不是整图缩放器。
6. 用户在拖拽已选中分镜的图片时，能立刻看到裁切内容实时更新。
7. 用户能在漫画模式下把图片粘贴进选中或悬停分镜，在 CG 模式下粘贴成新的全 stage 分镜。
8. 用户能从右侧 Inspector 调整文字排版与气泡形状/样式。
9. 用户能同时拖动气泡尾尖和尾根连接点，而不会破坏气泡外轮廓。
10. 用户能相信：手动保存、返回首页、切换项目或关闭页面后，不会丢失重要工作。
## 11. 内置创作辅助 Agent / Built-in Creator Assistance Agent

Use the top-right `Inspector` / `Agent` sidebar toggle to open the built-in creator assistance Agent. This feature helps the creator inspect the current page, consider options, and prepare bounded editor actions. Manga creation remains the creator's work: the Agent should assist with suggestions and mechanical edits, not replace the creator's story, composition, dialogue, or artistic judgment.

This product feature is for the current MangaMaker project; it is not the external coding-agent workflow used to modify the repository.

Use the ribbon workspace buttons to switch the center area between `Comic` and `Docs`. `Comic` shows the page canvas. `Docs` first shows a file-viewer list of Markdown file names. Right-click in the Docs workspace to create a Markdown document, or right-click any file row to rename or delete that document. Click a file such as `Producer.md` to open its rendered Markdown document; use the explicit `Edit` action only when you need to change raw Markdown. The right Agent sidebar can stay open in either mode. The left sidebar remains a comic-page overview only: it shows page thumbnails and page operations, not document overviews. The project documents are persisted under the project `docs/` folder and are the reliable production record; chat is only conversation. Every project must contain `docs/PrimeDirective.md`; this file defines whether the project is manga, CG, illustrated light novel, or another concrete form, and the Agent treats it as project-level direction. Default role metadocs live under `docs/roles/` and their file names must match their role names, for example `docs/roles/Producer.md`. Default role documents are starter files; if you delete one, MangaMaker must not silently recreate it. `PrimeDirective.md` is the exception: it is required and cannot be deleted.

Each active Agent role has one metadoc, one working directory, and one Conversation Context session in the current project. `PrimeDirective.md` defines the whole project; the metadoc defines that role only; the role working directory stores that role's durable output. Default metadocs include the assistant metadoc, production plan, story architecture, storyboard overview, script/dialogue, art supervision, continuity check, and image prompts. You can create ordinary Markdown documents that are not bound to any role. In the Agent sidebar, create a new role by setting the role name, choosing or creating its metadoc, and confirming its working directory under `docs/`. If you delete a metadoc, its role disappears. If you delete a role from the Agent sidebar, the metadoc stays in Docs as an ordinary Markdown document.

The Agent sidebar shows configuration before you chat:

- Test mode: set `MANGAMAKER_AGENT_TEST_MODE=1` for deterministic local tests.
- OpenRouter web backend: set `OPENROUTER_API_KEY` and explicitly set `MANGAMAKER_AGENT_MODEL`.
- Context window: `MANGAMAKER_AGENT_CONTEXT_WINDOW_TOKENS` controls the Agent prompt/input budget and defaults to `262144` for Kimi K2.6. You can override it per project from `Agent Config`; lowering it makes the Agent compact old conversation and tool results more aggressively.
- Optional OpenRouter tuning: `MANGAMAKER_AGENT_MAX_OUTPUT_TOKENS` defaults to `16384`, `MANGAMAKER_AGENT_REASONING_MAX_TOKENS` defaults to `2048`, `MANGAMAKER_AGENT_REASONING_EXCLUDE` defaults to `true`, `MANGAMAKER_AGENT_TEMPERATURE` defaults to `0.1`, and `MANGAMAKER_AGENT_TOP_P` defaults to `0.9`. The reasoning cap prevents Kimi-style reasoning models from exhausting the output budget before returning Agent JSON.
- The built-in Agent accepts DeepSeek or Kimi/Moonshot OpenRouter models with explicit capability labels. Multimodal choices must support image input, text output, and JSON `response_format`. `deepseek/deepseek-v4-pro` is the allowed text-only document model: it cannot see images, pages, or renders, but it can read/write Markdown documents while using `PrimeDirective.md` and the active role metadoc as pinned context. Google, Anthropic, OpenAI, and other text-only models are not offered as valid built-in Agent choices.
- Never commit real API keys or local secret files.
- The sidebar states whether visual input is enabled. If the configured model is text-only or a screenshot send fails, it shows a warning instead of pretending the Agent saw the page.
- Open `Agent Config` when you need manual steering. You can edit the system prompt for future turns, set the context-window token budget, and edit the conversation context sent to the model, including your own prior instructions and the Agent's prior replies. MangaMaker still enforces the JSON response protocol, command schema validation, and confirmation rules.

Assistance behavior:

- The Agent can suggest improvements, critique page readability, point out context details, and update Markdown documents. It does not directly execute panel, text, bubble, layer, page, or save operations.
- The Agent can work in role modes such as Assistant, Producer, Director, Storyboard Designer, Script Designer, Art Supervisor, Continuity Supervisor, Prompt Engineer, or creator-defined roles. Each active role uses its metadoc as the role prompt and role definition, and uses its working directory for durable output. Creating a role asks for the role name, metadoc, and working directory; edit that role's metadoc when you want to change its prompt. These roles are there to help maintain project documents and prepare scoped actions; they do not replace the human creator.
- The Agent reads the project through a tool-style harness. Multimodal models get an intentionally small initial context: project summary, page index, current-page marker, current selection, `PrimeDirective.md`, active role metadoc, active role working directory, and available tool definitions. Text-only document models get `PrimeDirective.md`, the active role metadoc, active role working directory, and Markdown document tools, but no page/image/render tools.
- The Agent does not receive every page object, every image asset, every Markdown document, or the current screenshot up front. It should search with `searchProject`, inspect specific pages with `readPage`/`readPages`, list relevant resources with `listImageAssets`, inspect role bindings with `listRoles`, read documents with `listDocuments`/`readDocument`/`searchDocuments`, and request renders only when visual judgment is needed.
- When the Agent changes durable production direction, it should update Markdown through document tools instead of relying on chat memory. It should use `appendDocument` for additive notes, `replaceDocumentSection` for heading-based section updates, `replaceDocumentText` for exact small replacements, and `writeDocument` only for full-document creation or replacement.
- When the Agent needs to judge the final look of a page, it can request `renderPage` or `renderPages`. MangaMaker then gives it both the composed page screenshot(s) and the matching structured resources.
- To reduce multimodal token cost, the Agent should inspect structured resources first, use `renderPages` only for small `detail="preview"` samples, and use `renderPage` with `crop` when it needs a local region. `detail="detail"` is for small text, faces, or fine line art only. PNG versus JPG is not the main cost lever; fewer pixels, fewer images, and tighter crops are.
- Repeated identical tool requests are skipped, and each turn has a bounded tool budget. If the budget is reached, the Agent pauses instead of answering from incomplete evidence, then asks you to Continue or Stop.
- For document restructuring tasks, the Agent should read the target Markdown document once, then use the narrowest document mutation tool that can persist the change. If it repeatedly asks to read the same document, MangaMaker skips the duplicate request and then pauses the run rather than allowing an infinite loop.
- The Agent sees image assets as resource references and bounded on-demand visual renders rather than receiving every original image as unbounded base64 prompt text.
- Conversation Context is saved per project and per role by default and is the only editable history/context surface for the active role. Switching roles switches to that role's one session. Open `Agent Config` to edit, add, delete, or `Clear context` entries before the next turn. `Clear context` clears only the active role's session. The stable role-indexed record is `agent-conversation-context.json` in the project folder, so local automation and other debugging agents can read all role sessions directly instead of scraping browser storage.
- Tool call status appears inside the Agent conversation flow, not in a separate Tool Log panel. Pressing `Clear context` clears the visible conversation and tool status entries without adding a delete event to the tool timeline.
- Each turn is saved as an `agentRun` with steps such as `model_request`, `tool_call`, `tool_result`, `model_resume`, `retry`, and `command_plan`. The sidebar listens to the run through Server-Sent Events, so long model calls show live state and do not depend on one blocking chat fetch. Document writes requested by the Agent must include an `operationId`; retries with the same id and edit are safe, while conflicting reuse is rejected.
- The creator decides whether to apply the Agent's suggestions manually in the editor.
- Page-edit requests should be recorded as document instructions or checklist items, not as executable command plans.

Desktop production builds call Tauri Agent commands instead of web-only fetch endpoints. If no native Agent backend is configured, the chat input is disabled and the sidebar explains that the desktop Agent backend is unavailable.

For live debugging while the Agent sidebar is open, use `GET /__mangamaker__/agent/debug` from the web/Vite runtime or run `window.mangaMaker.agent.getDebugSnapshot()` in the browser console. The debug snapshot includes the active run id and status when available. To inspect a run directly, use `GET /__mangamaker__/agent/runs/:runId?projectId=...` or subscribe to `GET /__mangamaker__/agent/runs/:runId/events`. For durable Conversation Context, read `projects/<project-folder>/agent-conversation-context.json` or use `GET /__mangamaker__/agent/conversation-context?projectId=...&roleId=...`. The debug snapshot shows busy state, pending tool call, recent messages, sanitized context summary, and recent tool logs. It does not expose API keys or raw base64 canvas screenshots.
